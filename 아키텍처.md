# 법률 챗봇 서비스 아키텍처

## 1. 아키텍처 선택
본 서비스는 초기 구현 속도와 실용성을 고려하여 1번 아키텍처인 RAG 기반 구조를 채택한다.

이 구조는 사용자의 질문을 받으면 먼저 관련 법률 문서를 검색하고, 검색된 내용을 바탕으로 모델이 답변을 생성하는 방식이다. 이를 통해 법률 근거를 함께 제시할 수 있고, 단순 생성형 응답보다 더 안정적인 답변이 가능하다.

## 2. 핵심 목표
- 사용자가 법률 관련 질문을 입력하면 답변을 제공한다.
- 답변과 함께 예방 안내, 주의사항, 추가 확인 포인트를 제공한다.
- 법률 자문을 대체하지 않고, 사전 예방형 보조 서비스로 활용되도록 한다.

## 3. 사용 언어 및 기술 스택
- 백엔드 언어: Python
- 프론트엔드 언어: JavaScript
- 웹 서버: FastAPI
- 프론트엔드 프레임워크: React
- LLM: EXAONE/EXAONE-4.0-1.2B
- 임베딩 모델: BAAI/bge-m3
- 벡터 데이터베이스: FAISS
- 모델 실행: Hugging Face Transformers
- 검색 방식: 임베딩 기반 유사도 검색, 키워드 검색 보조 활용
- 데이터 처리: Python 기반 전처리 및 문서 청크 분할

## 4. 아키텍처 구성 요소
### 4.1 프론트엔드
- 사용자가 질문을 입력하고 답변을 확인하는 웹 화면을 담당한다.
- 채팅형 입력창, 답변 영역, 예방 안내 섹션을 제공한다.
- 사용자의 질문을 백엔드로 전달하고, 응답을 화면에 표시한다.

### 4.2 백엔드
- Python과 FastAPI로 구성된 API 서버이다.
- 사용자의 요청을 수신하고, 검색 모듈과 모델 모듈을 연동한다.
- 최종 응답을 정리하여 프론트엔드에 반환한다.

### 4.3 검색 모듈
- 민사법 데이터에서 질문과 관련된 문서나 조항을 찾는다.
- 검색 결과를 기반으로 모델이 더 정확한 답변을 생성할 수 있도록 지원한다.
- 검색 품질이 답변 품질에 직접적인 영향을 미친다.

### 4.4 모델 모듈
- EXAONE 모델을 로딩하고, 검색된 문맥을 바탕으로 답변을 생성한다.
- 법률 정보 제공과 함께 예방형 안내를 포함한 응답을 만든다.

### 4.5 응답 포맷팅 모듈
- 생성된 답변을 다음 형식으로 정리한다.
  - 핵심 답변
  - 관련 법률 근거
  - 주의사항
  - 예방 조치
  - 전문가 상담 권유

### 4.6 안전성 모듈
- 개인정보 입력에 대한 경고를 포함한다.
- 법률 자문을 대체하는 것이 아님을 명확히 한다.
- 고위험 상황에서는 전문가 상담을 권장한다.

## 5. 동작 흐름
1. 사용자가 웹 화면에서 질문을 입력한다.
2. 프론트엔드가 백엔드 API로 요청을 보낸다.
3. 백엔드가 검색 모듈을 실행해 관련 법률 내용을 찾는다.
4. 검색 결과를 모델에 전달한다.
5. 모델이 답변을 생성한다.
6. 백엔드가 응답을 정리해 프론트엔드로 반환한다.
7. 사용자는 답변과 예방 안내를 확인한다.

## 6. 장점
- 초기 구현 속도가 빠르다.
- 검색 기반 구조라 답변의 근거를 함께 제시하기 쉽다.
- 환각 현상을 줄이는 데 유리하다.
- 사전예방형 안내를 넣기 좋다.

## 7. 단점
- 검색 품질이 답변 품질에 큰 영향을 미친다.
- 데이터 정제와 문서 분할 작업이 중요하다.
- 모델이 과도하게 단정적인 답변을 하지 않도록 제어가 필요하다.

## 8. 한 줄 요약
이 아키텍처는 “질문 → 관련 법률 검색 → 모델 답변 생성 → 예방 안내 포함” 흐름으로, 법률 챗봇 서비스의 초기 구현에 가장 적합한 구조이다.
